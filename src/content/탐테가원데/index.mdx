<ThreadItem>

  테스트를 통해 **학습** 한 것을 활용함
</ThreadItem>

<ThreadItem>세션 시간을 정해놓고 **테스트 차터**로 진행함</ThreadItem>

<ThreadItem>
  테스트 차터는 대게 테스트 목적, 자원, 찾아야 하는 정보로 구성되지만 양식이
  어렵다면 간략하게 목적을 쓰는 것도 좋음! 예를 들어서 '사용자가 구매를 완료하지
  못하는 시나리오를 찾아라' 같은 것도 좋은 차터가 될 수 있음
</ThreadItem>

<ThreadItem>
  좋은차터란? 동작을 구체적으로 명시하지 않으면서 테스트가 나아가야할 방향을
  작성한 것
</ThreadItem>

  <ThreadItem>
  ```tsx
  // ❌ Don’t 
‘보안에 취약한 모든 부분을 찾기 …(생략) 탐험을 떠나자’ // 너무 광범위함
‘john doe를 가지는 이름을 제대로 처리하는지 …(생략) 탐험을 떠나자’ // 너무 구체적임
````
  ```tsx
  // ✅ Do
  // 여러 차터로 세밀하게 나누기
  ‘보안에 취약한 점을 찾기 위해서 …(생략) 탐험을 떠나자’
  ‘사용자가 구매하지 않을 콘텐츠에 접근할 수 있는지 …(생략) 탐험을 떠나자’
````

훌륭한 차터는 아주 명확한 행동 지침이나 결과물에 대한 언급 없어도 영감의 원천이 된다.

</ThreadItem>

<ThreadItem>
  차터 작성하는 방법! <br /> 1. 요구사항 회의 도중에 발견되는 사항. 예를들어
  개발자 얘기하면서 탐색해야할 곳을 발견하거나, 숨겨진 요구사항(신뢰성,
  규모가변성 등)들을 차터로 만들자 <br /> 2. 새로운 기능과 연동이 잘 되는지,
  변경된 조건에도 잘 동작하는지, 미래에 발생할 위험 요소(해킹 등)에 대해 차터를
  만들자 <br />
  3. 현재 집중하고 있는 차터에서 다른 방향으로 탐색하고 싶은 유혹이 들 때
</ThreadItem>
